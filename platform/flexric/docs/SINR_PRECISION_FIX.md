## 1. Problem Statement

When the xApp received KPM indication messages from the E2 nodes, all SINR values
appeared as whole-number integers (e.g. `+35.00 dB`, `+57.00 dB`) despite the ns-3
simulator computing full floating-point values internally.

**xApp KPM log (before fix):**
```
[04:38:21] TRACK  NEW UE 15 attached to Cell 8  SINR=+35.00 dB
[04:38:21] KPM    Serving  Cell  8  UE 15  SINR  +35.00 dB
[04:38:21] TRACK  NEW UE 16 attached to Cell 8  SINR=+40.00 dB
[04:38:21] TRACK  NEW UE 11 attached to Cell 6  SINR=+57.00 dB
[04:38:21] TRACK  NEW UE  1 attached to Cell 2  SINR=+50.00 dB
```

**Scenario internal log (same simulation moment):**
```
[KPM-REAL] UE 15 Cell 8   SINR=35.8815 dB
[KPM-REAL] UE 16 Cell 8   SINR=41.5096 dB
[KPM-REAL] UE 11 Cell 6   SINR=57.3829 dB
[KPM-REAL] UE  1 Cell 2   SINR=51.0131 dB
```

The fractional part was silently discarded at every KPM report, making all SINR values
floor-truncated integers delivered over the E2 interface.

---

## 2. Root Cause Analysis

### 2.1 The Truncation Chain

The SINR value travels through four layers between the ns-3 PHY and the xApp.
Precision was lost at **Step 2** — an implicit C++ `double → long` narrowing
conversion — and the integer was then copied verbatim into the ASN.1 REAL field.

```
Step 1  mmwave-enb-net-device.cc
        sinrThisCell = 10 * log10(m_l3sinrMap[imsi][cellId])
        → double  35.8815 dB   ✔  full precision here

Step 2  mmwave-enb-net-device.cc  line 1259-1260
        CreateL3RrcUeSpecificSinrServingWithRsrp(
            m_cellId, m_cellId, sinrThisCell, rsrpEncoded)
                                 ^^^^^^^^^^^^ double passed to long parameter
        → implicit cast: (long)35.8815 = 35   ✘  PRECISION LOST HERE

Step 3  asn1c-types.cc  AddSinr(long sinr)
        *m_measQuantityResults->sinr = 35      (SINR_Range_t = long*)

Step 4  kpm-indication.cc  line 826
        choice.real = *ssb->sinr               = 35.0
        → xApp receives 35.00 dB
```

**All relevant function signatures accepted `long sinr`**, causing an implicit
narrowing conversion at every call site that passed a `double`:

| Function | Old parameter | Effect |
|---|---|---|
| `MeasQuantityResultsWrap::AddSinr` | `long sinr` | truncates fractional part |
| `MeasResultEutra::AddSinr` | `long sinr` | truncates fractional part |
| `L3RrcMeasurements::CreateL3RrcUeSpecificSinrServing` | `long sinr` | truncates fractional part |
| `L3RrcMeasurements::CreateL3RrcUeSpecificSinrServingWithRsrp` | `long sinr` | truncates fractional part |
| `L3RrcMeasurements::AddNeighbourCellMeasurement` | `long sinr` | truncates fractional part |
| `L3RrcMeasurements::AddNeighbourCellMeasurementWithRsrp` | `long sinr` | truncates fractional part |

The underlying storage type `SINR_Range_t` is generated by the ASN.1 compiler as
`long` (3GPP TS 38.331 defines `SINR-Range ::= INTEGER (0..127)`). This type cannot
hold a `double` directly, so the fix uses a **fixed-point scale convention** — see
[Fix Strategy](#3-fix-strategy).

### 2.2 Why the Scenario Log Shows Full Precision

The `[KPM-REAL]` lines are printed inside `KpmHandoverCsvCb()` in the scenario,
which reads directly from the ns-3 `L3ReportUeSinr` trace — **bypassing the entire
E2/KPM ASN.1 encoding pipeline**. This is why the scenario always showed correct
values while the xApp received integers.

The SINR shared-file mechanism (`~/sinr_shared.csv`) also bypasses the pipeline for
the same reason: the scenario writes raw `double` values directly to CSV.

## 3. Fix Strategy

Because `SINR_Range_t` is an ASN.1-generated `long` type that cannot be changed
without modifying generated code, the fix uses a **fixed-point representation**:

```
Store  : sinr_long = round(sinr_dB * 10000)   →  35.8815 dB stored as 358815
Recover: sinr_dB   = sinr_long / 10000.0      →  358815 / 10000.0 = 35.8815 dB
```

This gives **4 decimal places of precision** (± 0.00005 dB error at most) while
remaining entirely within the existing `long` storage type. No ASN.1 schema changes
are required.

Three files were edited:

| # | File path | Role |
|---|---|---|
| 1 | `contrib/oran-interface/model/asn1c-types.h` | Change function signatures `long sinr` → `double sinr` |
| 2 | `contrib/oran-interface/model/asn1c-types.cc` | Store scaled fixed-point value |
| 3 | `contrib/oran-interface/model/kpm-indication.cc` | Divide by 10000.0 when writing ASN.1 REAL field |

## 5. Verification

After rebuilding and running the simulation, the xApp log immediately showed
full floating-point SINR values delivered through KPM:

```
[09:33:20] TRACK  UE 15 SINR update  Cell 8  +21.50 dB
[09:33:20] KPM    Serving  Cell  8  UE 15  SINR  +21.50 dB
[09:33:20] TRACK  UE 16 SINR update  Cell 8  +23.77 dB
[09:33:20] KPM    Serving  Cell  8  UE 16  SINR  +23.77 dB
[09:33:20] TRACK  UE 13 SINR update  Cell 7  +41.53 dB
[09:33:20] TRACK  UE 11 SINR update  Cell 6  +58.88 dB
[09:33:20] TRACK  UE 10 SINR update  Cell 5  +43.22 dB
```

SINR values now carry 2 decimal places of visible precision (the display format
rounds the 4-decimal stored value to 2 for readability), compared to the previous
integer-only `.00` values.

## 6. Impact on the AWF Algorithm

The AWF (Adaptive Weighting Function) handover algorithm in `xapp_es_with_cell_util`
uses SINR in three places, all of which benefit from this fix:

| Formula | Usage | Before fix | After fix |
|---|---|---|---|
| `f(γ) = (target_sinr − serving_sinr) / 60.0` (Eq. 5) | SINR quality factor in fWF score | Differences of 0 or ±1 when cells differ by < 1 dB | Accurate fractional difference |
| `f_sinr` feeds `Ωx` weights (Eq. 2–4) | Dynamic weight calculation | Noisy, coarse | Smooth, matches paper |
| NCL filter: `sinr ≥ MIN_SINR (−10 dB)` | Neighbour cell eligibility | Already integer-safe | Unchanged |

Additionally, because KPM-delivered SINR is now accurate, the SINR shared-file
fallback (`~/sinr_shared.csv`) is no longer the sole source of reliable SINR data.
Both paths now deliver equivalent precision.
